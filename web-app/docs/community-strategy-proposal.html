Community 內容分發與審核機制

在使用者仍少、內容以官方為主的階段，如何讓台灣使用者看到台灣的內容、讓好內容自然浮上來，並在開放發布時守住安全底線。

01

要解的問題

Community 的畫面已經做好，但背後沒有內容分發規則。兩個必須先回答的核心問題：

  • 在地化落差。假設我是台灣使用者，但平台上印度使用者很多——不能推薦一片印度 MV，要優先推台灣人想看的內容。
  • 冷啟動。新平台各別用戶還不多，一開始內容以官方居多，排序不能依賴大量互動數據。
  • 開放發布的安全。使用者一旦公開作品，需要機制擋下 NSFW 等違規內容——而且這個 App 會上傳人臉，風險不只色情。
02

在地化：locale-first，global 優雅補位

不走「只看 global」（就是印度 MV 淹沒台灣人的情境），也不走「只看 local」（早期在地內容太薄、feed 會空）。取兩者中間。

  • 每則內容打三個標籤：content_locale（語言）、origin_region（創作者地區）、is_official（官方內容）。
  • 主 feed 以使用者 locale 內容優先排序；在地內容不足時才用「官方 → 全球高分」回填，並設回填上限（外語內容不超過 feed 的 20–30%），永遠不會變成一片外語。
  • 另開 Global／探索 分頁給想看世界的人——把「看全球」變成主動選擇，而非預設塞給使用者。
  • 地區來源以 IP 為主（免設定、自動）；建議日後補「帳號設定可覆寫」以處理 VPN 與旅居使用者。

業界怎麼分——共通心法是把「人工精選 / 演算法 / 排行榜」當三種不同 surface，而非塞進同一條公式：

平台在地化邏輯對我們的啟發
Spotify編輯歌單分市場（人工）、演算法自然依語言分群、排行榜分國家/全球三種 surface 分離
Netflix「Top 10 in Taiwan」為地區榜；每一列（row）用不同演算法冷啟動靠 onboarding 偏好＋所在地熱門
TikTok用裝置語言/IP 先餵地區熱門，再用完播率快速個人化；新內容先小範圍本地測試「內容有出生地，表現好可外擴」
YouTube首頁個人化、Trending 分國家；偵測影片語言對齊使用者語言語言對齊是強訊號
03

內容演算法路線

分階段導入——早期純規則、零個人化；累積數據後才上個人化與真 trending。

PHASE 1

官方為主 · 純規則

冷啟動 · 現在
  • Feed 大部分為官方精選（可備多語版本解決在地化）
  • ADMS 人工排前 20（編輯 pin）
  • 第 21 名後用規則式排序公式自動排（見 §04）
  • locale 先過濾、IP 判定
PHASE 2

導入個人化

UGC 與互動數據足夠後
  • 協同過濾（「喜歡這支的也喜歡…」）
  • 內容標籤比對（mvType / genre / mood）
  • Trending 改看速度（近 N 小時增幅）
  • locale 由硬過濾放寬為排序權重
PHASE 3

完整 FYP

規模化後
  • 以看完率/停留優化的個人化推薦流
  • 探索 vs 熟悉的動態平衡
  • 創作者成長與分潤訊號納入
04

Phase 1 排序公式（第 21 名之後）

人工排不到的長尾要自動排。採時間衰減的加權互動分數：純看累積互動會讓榜單僵化、新人永遠出不了頭，加時間衰減能讓內容流動。

① 加權互動值 → ② 時間衰減
# ① 互動價值（依「意圖強度」加權：分享 > 讚 > 完播 > 播放） E = 1·plays + 2·completes + 4·likes + 10·shares # ② 時間衰減（Hacker News gravity 式） score = E / (age_hours + 2) ^ 1.8 # ③ 冷啟動平滑（避免「3 播放 2 讚」用比率灌水贏過 3000 播放） smoothed_like_rate = (likes + C·m) / (plays + C) m = 全站平均讚率 · C = 偽計數（如 50）
play ×1 complete ×2 like ×4 share ×10 gravity 1.8 offset +2h
  • 新內容探索位。feed 保留少量固定名額給發布 < 24h 的內容，給新創作者曝光、防止大者恆大。
  • 全參數可調。權重（1/2/4/10）、gravity、回填上限都設為營運可調設定，不改程式即可微調——ADMS 可承接這個控制面。
為什麼是這個公式

Hacker News 與 Reddit 的熱度排序都採「互動 ÷ 時間衰減」的形式；它可直接用一句 SQL 算出、無需 ML，正好適合 Phase 1 的資料量與工程成本。Phase 2 再用協同過濾疊上個人化。

05

審核機制：自動預審 + 人工覆核灰色地帶

不走純人工（不擴展、慢、審核員負擔），也不走純自動（誤判與法律風險）。內容按「發布」後先進審核，不直接公開。

送出發布
draft → submitted
自動預審
影格＋縮圖＋音訊＋文字
安全
自動放行
→ published
違規
自動擋下
→ rejected，通知使用者
灰色
進人工佇列
pending_review →（人工）→ 放行 / 拒絕
發布後檢舉
任何人 Report → 回到 pending_review 重審

分三個攔截點，難度差很多。我們是 AI 生成內容——能在輸入端（prompt）就擋，是比一般 UGC 平台更強的優勢：

元件難度怎麼做
① Prompt 文字預審低生成時把 prompt 丟文字 moderation API（OpenAI moderation 免費，或 Google/AWS），擋掉明顯違規描述。一個 API call。
② 影片自動預審中用 SaaS：Hive Moderation（UGC 影片專精、最準）、AWS Rekognition（在 AWS 最省事）、Google Video Intelligence、Azure Content Safety。回傳各類別分數。不建議自架開源模型。
③ 三向閾值路由低<低標→放行；>高標→擋；中間 → pending_review。純設定判斷。
④ 人工佇列後台中內部審核頁：待審清單＋影片預覽＋機器分數＋通過/拒絕鍵。建議直接擴充 ADMS，內容加 moderation_status 欄位。
⑤ 檢舉回流低–中每支 MV 加 Report 鍵 → 寫 report 記錄 → 重進佇列。

Phase 1 兩個選項——待主管依工程成本拍板：

建議 · 可直接擴展

完整版：①②③④ 全上

只有灰色地帶進人工，可無痛接到 Phase 2。整體難度中等，主要成本是串一個 SaaS API ＋一個簡單後台。長期審核成本最低。

最快上線

輕量版：①② + 全部人工放行

只自動擋明顯違規，其餘因 Phase 1 官方為主、UGC 量少 → 全部進人工放行。最快上線，量大後再補三向路由。

⚠ NSFW 不是唯一風險

這 App 會上傳人臉，審核類別必須擴大：肖像／deepfake 同意（有人上傳名人或他人的臉）、歌曲著作權、CSAM（法律強制通報、零容忍）。這些要一開始就納入政策與偵測類別，不能只擋色情。

06

需要的資料模型

支撐上述所有機制，最小欄位增補：

content.content_locale · origin_region · is_official
content.moderation_status // screening | approved | rejected | pending_review
content.moderation_scores (jsonb) · reviewed_by · reviewed_at
content.plays · completes · likes · shares · published_at
reports table // reporter · content_id · reason · status

排序分數建議離線批次預算（每 N 分鐘重算並快取），feed 讀快取即可，避免每次請求即時計算。

07

待主管拍板

Q1
Phase 1 審核走完整版還是輕量版？

取決於是否願意在 Phase 1 就投入一個 SaaS 審核 API ＋後台的整合成本。

Q2
影片審核供應商？

建議 Hive（最準）或 AWS Rekognition（若已在 AWS）。需確認預算與每月審核量級。

Q3
肖像／deepfake 同意政策由誰定？

牽涉法務——上傳他人臉的規範與同意流程，需與法務對齊後才能開放發布。

Q4
ADMS 是否承接排序控制面與審核佇列？

建議沿用 ADMS 擴充，避免另建後台；需確認 ADMS 團隊可排期。
