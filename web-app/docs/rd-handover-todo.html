§1 程式 Bug / 技術債

產品化(接真實 backend)前需要處理的程式問題。部分經 PM 決策後暫不動程式、僅列此追蹤。

使用者媒體是 blob URL + 前端生成 ID,直接進 API requestP0RD

MvRoom.importAudio／addPhotoFromFile 把 URL.createObjectURL(...) 的 blob: URL 與 crypto.randomUUID() 塞進 Song.url／CharacterPhoto.url,原樣送進 MvCreateRequest。Schema 只是 z.string(),看起來「一個字串就夠」,但真實 backend 無法解參照別台機器的 blob URL。

為何重要:需要先設計「上傳端點」→ 拿到伺服器 URL 再送 create,contract 目前沒有這個掛勾。另外 object URL 從未 revokeObjectURL,有緩慢記憶體洩漏。

mv/MvRoom.tsx:61-76lib/api/schemas.ts
輪詢(pollJob)一次抖動就當作失敗、且為 demo 速度P1RD

任何一次 fetch throw 就停止輪詢並觸發 onError → 被 provider 轉成 failed + markFailed,一個 Wi-Fi 抖動就會「失敗」一個伺服器端仍在跑的 job。沒有重試預算、沒有 backoff;DEFAULT_INTERVAL_MS = 120 對真後端等於每秒約 8 次請求;且 onError 丟掉了 job 本身的錯誤細節(contract 也沒有 job 的 error 欄位)。

產品化必改:加入重試/backoff、區分「job 失敗」與「輪詢失敗」、把輪詢間隔調到合理值、contract 補 error 欄位。
